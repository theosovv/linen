

# Linen: Функционально-Реактивный Язык Музыкального Программирования

## 1. Архитектура Языка

### 1.1 Парадигма и Типизация

#### 1.1.1 Функционально-реактивное программирование (FRP)

Функционально-реактивное программирование (FRP) лежит в основе архитектуры Linen, обеспечивая декларативный подход к моделированию временных зависимостей в аудиоприложениях. В отличие от императивных систем, где состояние изменяется через явные присваивания, FRP оперирует двумя фундаментальными абстракциями: **поведениями (behaviors)** — непрерывными, временно зависимыми значениями, и **событиями (events)** — дискретными моментами во времени . Поведение семантически представляется как функция `Behavior a = Time → a`, что позволяет трактовать аудиосигналы как математические объекты, независимые от конкретной реализации.

Ключевое преимущество FRP для музыкального программирования заключается в **возможности модифицировать программу без потери состояния существующих аудиоузлов**. Традиционные императивные языки вроде SuperCollider требуют либо явного именования узлов для последующей модификации, либо полной переоценки программы со сбросом всех состояний . Инкрементальное функционально-реактивное программирование (IFRP) решает эту проблему через механизм сопоставления состояний между ревизиями программы: при редактировании кода система вычисляет разницу между старым и новым графом, переносит состояние совпадающих узлов и инициализирует новые узлы значениями по умолчанию . Это критично для live coding, где музыкант ожидает мгновенной реакции на изменение кода без щелчков или выпадения нот.

Стрелочный подход к FRP, реализованный в Yampa, предоставляет дополнительные гарантии эффективности . Вместо прямого манипулирования сигналами как первоклассными значениями, программист работает с **сигнальными функциями (signal functions)** `SF a b`, явно представляющими преобразования входных сигналов в выходные. Это предотвращает утечки времени и пространства, характерные для наивных реализаций FRP, где случайное сохранение ссылки на старый сигнал приводит к накоплению истории . Для Linen выбран гибридный подход: высокоуровневое описание сигналов как behaviors для удобства, с автоматическим преобразованием в сигнальные функции для оптимизации.

| Подход FRP | Представление сигнала | Преимущества | Недостатки |
|-----------|----------------------|------------|-----------|
| Классическое (Fran) | `Signal a = Time → a` | Математическая чистота, простота | Утечки времени/пространства |
| Стрелочное (Yampa) | `SF a b` — сигнальная функция | Эффективность, предсказуемость | Более многословное |
| Инкрементальное (IFRP) | Помеченные термы с миграцией состояния | Hot reload без потери состояния | Сложность реализации |
| Реактивное (React) | Потоки событий + производные состояния | Интеграция с UI-фреймворками | Менее естественно для аудио |

#### 1.1.2 Система типов Хиндли-Милнера с расширениями

Система типов Хиндли-Милнера обеспечивает **полиморфизм параметрический с автоматическим выводом типов**, что критично для удобства музыкального программирования. Композитор может писать высокоуровневые преобразования без явных аннотаций, при этом сохраняя строгие гарантии корректности на этапе компиляции. Базовая система HM позволяет выражать типы вида `forall a. a → a` для полиморфных функций, но для аудиодомена требуются целенаправленные расширения.

**Расширения типовой системы Linen:**

| Расширение | Назначение | Пример применения |
|-----------|-----------|-----------------|
| Типовые классы | Перегрузка операций над сигналами | `class Num a ⇒ SignalNum a where (+), (*), negate` |
| Ранжированные типы | Хранение полиморфных функций в структурах данных | `data SF a b = ∀s. SF (s → a → (s, b))` |
| GADTs | Типобезопасные аудиографы | `data Node a where Osc ∷ Frequency → Node Signal` |
| Линейные типы | Управление ресурсами реального времени | `linear ∷ AudioBuffer ⊸ (AudioBuffer ⊸ a) → a` |
| Зависимые типы (ограниченные) | Размеры буферов в типах | `Vec ∷ Nat → Type → Type` |

Линейные типы особенно важны для аудиопрограммирования, поскольку позволяют **статически гарантировать корректное управление ресурсами** — например, что аудиобуфер будет освобождён ровно один раз после обработки. Это предотвращает утечки памяти в системах реального времени, где сборка мусора недопустима. Зависимые типы для размеров позволяют отлавливать ошибки несовместимости размеров буферов на этапе компиляции, а не во время выполнения.

Типизация временных характеристик — уникальное требование Linen. Сигналы параметризуются метаданными о временной шкале: `Signal (Rate 48000) Float` несовместим с `Signal (Rate 44100) Float` без явного преобразования. Это предотвращает тонкие ошибки, связанные с несовпадением частот дискретизации в сложных патчах.

#### 1.1.3 Разделение чистых функций и эффектов

Фундаментальный принцип функционального программирования — **разделение чистых функций и эффектов** — приобретает особое значение в аудиоконтексте. Чистые функции над сигналами (сложение, фильтрация, модуляция) могут быть свободно оптимизированы, мемоизированы, распараллелены без риска нарушения семантики. Эффекты же — взаимодействие с аудиодрайвером, чтение MIDI, запись файлов — требуют строгой изоляции.

В Linen эффекты инкапсулируются в специализированную монаду `Audio`:

```haskell
newtype Audio a = Audio { runAudio ∷ AudioContext → IO a }

-- Ключевые операции
readWav    ∷ FilePath → Audio (Behavior [Float])
writeWav   ∷ FilePath → Behavior [Float] → Audio ()
midiIn     ∷ Audio (Event MidiMessage)
audioOut   ∷ Behavior [Float] → Audio ()
```

Критическое требование — **предсказуемость времени выполнения эффектов**. В традиционных ленивых языках момент фактического выполнения может быть неочевиден; для Linen это неприемлемо. Запись в аудиобуфер должна происходить строго в момент вызова аудиоколбэка от драйвера. Поэтому система эффектов Linen **строгая по времени выполнения**, даже если вычисления внутри эффекта могут быть ленивыми.

### 1.2 Ядро языка: Аудио-примитивы

#### 1.2.1 Сигналы как функции времени

Фундаментальная абстракция Linen — **сигнал как функция от времени**. Математически `Signal a ≈ Time → a`, где `Time` — непрерывная величина, обычно `Double` в секундах. На уровне реализации сигналы дискретизируются с частотой дискретизации аудиосистемы, но это различие скрыто от программиста.

Важное различие — между **аудио-частотой (audio rate)** и **управляющей частотой (control rate)**. Сигналы могут работать на разных скоростях: аудиосигналы на 44100+ Гц, огибающие и LFO на 1000-4000 Гц, паттерны и секвенции на сотнях Гц. Linen поддерживает полиморфность по скорости через параметр типа: `Signal (Rate r) a`, с автоматической передискретизацией при необходимости.

Композиция сигналов естественна через аппликативный интерфейс. **Амплитудная модуляция**: `liftA2 (*) carrier modulator`. **Частотная модуляция**: `fmOsc (baseFreq + modIndex * modulator)`. Сложение сигналов — микширование. Всё это выражается без явных циклов или индексов, благодаря поднятию операций над сигналами.

#### 1.2.2 Осцилляторы: синус, пила, квадрат, треугольник

Ядро Linen предоставляет четыре фундаментальных осциллятора, реализованных через **фазовый аккумулятор** — переменное состояние, линейно нарастающее с обёртыванием для создания периодичности .

| Осциллятор | Математическое определение | Спектральное содержание | Особенности реализации |
|-----------|---------------------------|------------------------|----------------------|
| **Синус** | `sin(2π · phase)` | Фундаментальная только | Табличный lookup с интерполяцией или полиномиальная аппроксимация |
| **Пила** | `2·(phase mod 1) - 1` | Все гармоники, спад 6 дБ/окт | Требует bandlimited-реализации (BLEP) для музыкального качества |
| **Квадрат** | `sign(sin(2π · phase))` | Нечётные гармоники, 1/n | PWM через смещение порога сравнения |
| **Треугольник** | `1 - 4·\|phase - 0.5\|` | Нечётные гармоники, 1/n² | Естественно более мягкий, менее подвержен алиасингу |

Фазовый аккумулятор обновляется как `phase[n+1] = (phase[n] + Δφ) mod 1`, где `Δφ = frequency / sampleRate`. **Критически важно сохранение фазы при изменении параметров** — это обеспечивает непрерывность звука при инкрементальных обновлениях программы . При hot reload осциллятор продолжает с текущей фазы, а не перезапускается.

Для эффективности используется **табличный синтез с линейной или кубической интерполяцией**. Таблица размера 1024-4096 элементов обеспечивает соотношение SNR > 90 дБ при минимальных вычислительных затратах. Размер таблицы — параметр типа, позволяющий балансировать качество и скорость.

#### 1.2.3 Формы волны и фазовые аккумуляторы

Помимо стандартных осцилляторов, Linen поддерживает **произвольные формы волны** двумя способами: табличный (wavetable) и процедурный синтез.

**Табличный синтез** использует предвычисленный массив отсчётов с интерполяцией при чтении. Тип `Wavetable a = Array Int a` с операциями:

```haskell
lookupWavetable   ∷ Wavetable a → Phase → a           -- без интерполяции
lookupWavetableI  ∷ Wavetable a → Phase → a           -- линейная интерполяция
lookupWavetableC  ∷ Wavetable a → Phase → a           -- кубическая интерполяция
```

Морфинг между таблицами — `morphWavetables ∷ Signal Float → [Wavetable a] → Phasor → Signal a` — позволяет динамически менять тембр, интерполируя между несколькими формами волн.

**Процедурный синтез** определяет форму волны как функцию: `Waveform a = Phase → a`. Это даёт неограниченную гибкость за счёт вычислительных затрат. Примеры: физически моделированные формы волн, фрактальные структуры, генетические алгоритмы.

Фазовая модуляция (PM/FM) реализуется через сложение фаз: `pmOsc carrierFreq modulator index = sin(2π · (carrierPhase + index · modulator))`. Это эквивалентно FM при малых индексах, но вычислительно проще — не требует интегрирования модулирующего сигнала.

#### 1.2.4 ADSR-огибающие

ADSR (Attack, Decay, Sustain, Release) — стандартная модель огибающей для управления амплитудой или другими параметрами. В FRP-контексте ADSR — **сигнальная функция с внутренним конечным автоматом состояний**, управляемая событиями note-on/note-off.

| Фаза | Условие входа | Выход | Кривизна |
|------|-------------|-------|---------|
| **Idle** | Инициализация | 0 | — |
| **Attack** | Событие note-on | Интерполяция 0 → 1 за время attack | Линейная, экспоненциальная или сигмоидная |
| **Decay** | Уровень достиг 1.0 | Падение к sustain-уровню за время decay | Обычно экспоненциальная |
| **Sustain** | Уровень достиг sustain | Постоянный уровень | — |
| **Release** | Событие note-off | Падение к 0 за время release | Экспоненциальная |

Тип в Linen: `adsr ∷ ADSRParams → Event () → Event () → Signal Level`, где `ADSRParams = { attack, decay, sustain, release ∷ Time, curves ∷ CurveShape }`. Все параметры могут быть **сигналами**, а не константами — например, velocity нажатия клавиши может модулировать время attack.

**Инкрементальность критична**: при изменении параметров ADSR во время воспроизведения (например, укорочении attack time) текущая фаза и уровень сохраняются. Если звук находится в decay, он продолжает decay с новыми параметрами, а не перезапускается .

#### 1.2.5 Работа с временем: BPM, темп, квантизация

Музыкальное время в Linen **многослойно**: абсолютное время (секунды, отсчёты), музыкальное время (такты, доли), и относительное время (интервалы, задержки).

**Базовые типы времени:**

| Тип | Представление | Операции |
|-----|-------------|---------|
| `AbsoluteTime` | Секунды как `Double` | Сложение, сравнение, конвертация в отсчёты |
| `MusicalTime` | `(Bars, Beats, Fraction)` | Арифметика с переносом через размер такта |
| `Samples` | `Int64` — номер отсчёта | Точная синхронизация с аудиооборудованием |
| `Duration` | Относительное время | Сложение, умножение на скаляр |

Конвертация: `musicalToAbsolute ∷ BPM → TimeSignature → MusicalTime → AbsoluteTime`. При постоянном BPM это линейное соотношение; при модулируемом BPM — интегрирование.

**Квантизация** привязывает события к музыкальной сетке: `quantize ∷ Duration → Event a → Event a`. Шаг квантизации типично 1/4, 1/8, 1/16 или триоли. Для "человечности" предусмотрен `humanize ∷ Float → Event a → IO (Event a)`, добавляющий случайное смещение в заданном диапазоне.

**Транспорт** управляет глобальным временем: `play`, `stop`, `seek`, `setBPM`. Тип `Transport = Signal TransportState`, где `TransportState = { position ∷ MusicalTime, playing ∷ Bool, bpm ∷ BPM }`. Транспорт — единственный источник мутабельного состояния в системе; все производные сигналы чисты.

### 1.3 Реактивная модель

#### 1.3.1 Поведения (Behaviors): непрерывные сигналы

Поведение — **непрерывно определённое значение**, имеющее смысл в каждый момент времени. Семантически `Behavior a = Time → a`, но реализация использует ленивое вычисление и кэширование для эффективности.

**Ключевые операторы:**

| Оператор | Тип | Семантика |
|---------|-----|----------|
| `pure` | `a → Behavior a` | Константное поведение |
| `<*>` | `Behavior (a → b) → Behavior a → Behavior b` | Поточечное применение |
| `liftA2` | `(a → b → c) → Behavior a → Behavior b → Behavior c` | Бинарная операция над сигналами |
| `sample` | `Behavior a → Event Time → Event a` | Дискретизация в моменты событий |
| `integral` | `Behavior Double → Behavior Double` | Интегрирование по времени |

Интегрирование критично для физического моделирования: колебания пружин, электрические цепи, движение под действием сил. Реализация использует метод трапеций или Рунге-Кутты с адаптивным шагом для control-rate сигналов.

#### 1.3.2 События (Events): дискретные триггеры

Событие — **дискретное происшествие в определённый момент времени**, несущее значение. Тип `Event a = [(Time, a)]` — потенциально бесконечный список временных меток со значениями. Между occurrences событие не определено.

**Комбинаторы событий:**

| Комбинатор | Тип | Описание |
|-----------|-----|---------|
| `never` | `Event a` | Пустое событие |
| `merge` | `Event a → Event a → Event a` | Объединение потоков (приоритет левому при одновременности) |
| `filterE` | `(a → Bool) → Event a → Event a` | Фильтрация по предикату |
| `mapE` | `(a → b) → Event a → Event b` | Преобразование значений |
| `snapshot` | `Event a → Behavior b → Event (a, b)` | Выборка поведения в момент события |
| `gate` | `Event a → Behavior Bool → Event a` | Пропуск только когда условие истинно |

Преобразование событий в поведения — `hold ∷ a → Event a → Behavior a` (сохраняет последнее значение), `step ∷ Event a → Behavior a` (неопределённо до первого события), `accum ∷ a → Event (a → a) → Behavior a` (аккумуляция функций-обновлений).

#### 1.3.3 Сигнальные функции (Signal Functions) и композиция

Сигнальная функция `SF a b` — **преобразование входного сигнала в выходной с возможным внутренним состоянием**. Это абстракция стрелок (Arrows), обеспечивающая явное отслеживание потока данных и предотвращающая утечки .

**Базовые комбинаторы стрелок:**

| Комбинатор | Тип | Описание |
|-----------|-----|---------|
| `arr` | `(a → b) → SF a b` | Чистая функция как сигнальная функция |
| `>>>` | `SF a b → SF b c → SF a c` | Последовательная композиция |
| `&&&` | `SF a b → SF a c → SF a (b, c)` | Разветвление: один вход, два выхода |
| `***` | `SF a b → SF c d → SF (a, c) (b, d)` | Параллельная композиция |
| `loop` | `SF (a, c) (b, c) → SF a b` | Обратная связь (feedback) |

Обратная связь критична для рекурсивных аудиопроцессов: фильтры с обратной связью, ревербераторы, фазовые модуляторы. Пример: `lpfWithResonance = loop $ proc (input, feedback) → do ...`.

Стрелочная нотация позволяет писать в императивном стиле:

```haskell
synth = proc midiMsg → do
  freq  ← midiToFreq -< midiMsg
  osc   ← sinOsc -< freq
  env   ← adsr 0.1 0.3 0.7 0.5 -< gateFromMidi midiMsg
  returnA -< osc * env
```

#### 1.3.4 Инкрементальные обновления графа без потери состояния

Центральная инновация Linen — **hot reload с сохранением состояния**. Механизм IFRP  работает через:

1. **Метки термов**: каждый stateful-примитив в программе помечается уникальной меткой (явной или производной от позиции в AST)
2. **Сопоставление ревизий**: при изменении кода система строит отображение `oldLabel → newLabel` на основе структурного сходства
3. **Миграция состояния**: для сопоставленных пар состояние копируется; для новых узлов — инициализация по умолчанию
4. **Плавная интерполяция**: для изменившихся параметров применяется кроссфейд за ~10 мс во избежание щелчков

Пример: программист меняет `(sin-lfo 440)` на `(sin-lfo 880)`. Система распознаёт ту же структуру с другим параметром, сохраняет фазу LFO, меняет частоту с плавным переходом.

### 1.4 Структуры данных

#### 1.4.1 Списки и потоки для аудиосэмплов

Аудиосигналы естественно представляются как **потоки (streams)** — потенциально бесконечные, лениво вычисляемые последовательности. Тип `Stream a` с операциями `head`, `tail`, `cons`, `mapS`, `zipWithS`.

| Структура | Характеристика | Применение |
|----------|---------------|-----------|
| `Stream a` | Ленивый, бесконечный | Генерация, обработка в реальном времени |
| `Vector a` | Строгий, конечный, O(1) индексация | Загруженные сэмплы, таблицы волн |
| `Signal a` | Временная семантика, дискретизованный | Аудиосигналы, управляющие сигналы |
| `Buffer a` | Кольцевой, фиксированный размер | Линии задержки, реверберация |

Блоковая обработка эффективнее поэлементной: `chunk ∷ Int → Stream a → [[a]]` извлекает блоки для векторизованной обработки. JIT-компилятор разворачивает обработку блоков в эффективный машинный код.

#### 1.4.2 Графы для маршрутизации сигналов

Сложные аудиопатчи требуют **произвольной топологии**: параллельные эффекты, посылы на шины, обратная связь. Тип `SignalGraph i o` представляет направленный граф с входами типа `i` и выходами типа `o`.

```haskell
data SignalGraph i o where
  Node   ∷ Processor i o → SignalGraph i o
  Serial ∷ SignalGraph a b → SignalGraph b c → SignalGraph a c
  Parallel ∷ SignalGraph a c → SignalGraph b d → SignalGraph (a, b) (c, d)
  Feedback ∷ SignalGraph (i, f) (o, f) → SignalGraph i o  -- требует задержки в петле
```

Топологическая сортировка определяет порядок вычислений. Циклы требуют явной задержки: каждая петля обратной связи должна содержать хотя бы один элемент `delay ∷ Time → Signal a → Signal a`. Статический анализ проверяет это условие.

#### 1.4.3 Ленивые бесконечные структуры

Музыкальные структуры — паттерны, мелодии, ритмы — **концептуально бесконечны**. Ленивость позволяет определять их рекурсивно без немедленного вычисления.

Пример: фрактальная мелодия
```haskell
fractalMelody ∷ Melody
fractalMelody = base <> stretch 2 fractalMelody  -- <> — конкатенация
```

Для реального времени ленивость требует **контролируемой строгости**: аудиопримитивы вычисляются строго, высокоуровневые структуры — лениво. Оператор `force ∷ Stream a → IO ()` принудительно вычисляет элементы для профилирования и отладки.

## 2. Реализация: VM + JIT

### 2.1 Виртуальная машина

#### 2.1.1 Байткод для аудио-примитивов

VM Linen — **специализированная стековая машина** с регистровым окном для аудиообработки. Байткод оптимизирован для DSP: частые операции имеют однобайтовые опкоды, примитивы работают с блоками сэмплов.

| Категория | Опкоды | Описание |
|----------|--------|---------|
| Стековые | `PUSH`, `POP`, `DUP`, `SWAP` | Базовые операции |
| Арифметика | `FADD`, `FMUL`, `FSIN`, `FEXP` | Плавающая точка, скаляр и SIMD |
| Память | `LOAD`, `STORE`, `ALLOCA` | Локальные переменные, временные буферы |
| Аудио | `OSC_SIN`, `OSC_SAW`, `ENV_ADSR`, `BIQUAD` | Специализированные примитивы |
| Граф | `NODE_CREATE`, `CONNECT`, `DISCONNECT` | Манипуляция сигнальным графом |
| Поток | `YIELD`, `AWAIT`, `SCHEDULE` | Асинхронное выполнение |

Формат инструкции: 1 байт opcode + 4 бита режим адресации + 12-36 бит операнды. Регистровый файл: 32 плавающих регистра `f0-f31` (128-битные для AVX), 32 целых `i0-i31`.

Аудиопримитивы работают с **блоками по 64-256 сэмплов** — компромисс между кэш-эффективностью и задержкой. Внутри блока возможна векторизация; между блоками — синхронизация состояния.

#### 2.1.2 Управление памятью в реальном времени

**Сборка мусора в аудиопотоке недопустима** из-за непредсказуемых пауз. Linen использует комбинацию стратегий:

| Стратегия | Применение | Характеристики |
|----------|-----------|---------------|
| Региональное выделение | Аудиобуферы, временные структуры | O(1) аллокация, полное освобождение в конце кадра |
| Пулы объектов | Часто используемые структуры | Предварительное выделение, повторное использование |
| Линейные типы | Критичные ресурсы | Статическая гарантия единственного владения |
| Ручное управление | Линии задержки, большие буферы | Полный контроль, риск ошибок |

**Двухуровневая модель**: аудиопоток не аллоцирует вообще; контрольный поток (GUI, загрузка) использует обычный GC. Синхронизация через **lock-free очереди** (Michael-Scott queue) и кольцевые буферы.

#### 2.1.3 Избежание утечек времени и пространства

**Утечки времени** — рост вычислительной сложности с историей сигнала. Решение: все примитивы ядра имеют **O(1) сложность на отсчёт** с константной памятью. Интеграция — накопленная сумма, не полная история. FIR-фильтры — циклический буфер фиксированного размера.

**Утечки пространства** — удержание ненужных данных. Решения: линейные типы для одноразовых ресурсов, размерные типы для предотвращения накопления, периодическая принудительная оценка в non-real-time потоке.

Статический анализ вычисляет **верхние границы памяти** для каждой аудиофункции, гарантируя выполнение dead-line'ов.

### 2.2 JIT-компиляция

#### 2.2.1 Компиляция критичных по latency участков

JIT-компилятор транслирует горячие байткод-фрагменты в машинный код. **Триггеры компиляции:**

| Условие | Порог | Действие |
|--------|-------|---------|
| Частота вызова | > 1000 вызовов/сек | Фоновая компиляция |
| Время интерпретации | > 10% бюджета кадра | Приоритетная компиляция |
| Стабильность кода | > 5 секунд без изменений | Оптимизирующая компиляция |
| Явная аннотация | `@jit` | Немедленная компиляция |

**Многоуровневая система:**
- Уровень 0: Интерпретация с профилированием
- Уровень 1: Быстрая компиляция (Cranelift) — < 10 мс, базовые оптимизации
- Уровень 2: Оптимизирующая компиляция (LLVM) — в фоне, агрессивная векторизация

Компиляция выполняется в **отдельном потоке** с низким приоритетом. Готовый код атомарно подменяет интерпретируемую версию между кадрами.

#### 2.2.2 Генерация машинного кода для DSP-цепочек

DSP-цепочки — последовательности операций над сэмплами — основные кандидаты для оптимизации. **Ключевые трансформации:**

| Оптимизация | Описание | Выигрыш |
|------------|---------|--------|
| Векторизация (SIMD) | 4-16 сэмплов за инструкцию (AVX-512) | 4-16x ускорение |
| Развёртка циклов | Устранение оверхеда на индексацию | 10-30% |
| Инлайнинг | Устранение вызовов функций | 20-50% |
| Слияние операций | `a * b * c` → одна FMA-инструкция | Зависит от кода |
| Константная свёртка | Вычисление неизменяемых подвыражений | На этапе компиляции |

Пример: цепочка `sine 440 >>> lpf (sine 0.1 * 1000 + 2000) 0.7 >>> gain 0.5` компилируется в один цикл, где фаза LFO обновляется параллельно с вычислением фильтра.

Для полифонии используется **SOA (Structure of Arrays)** вместо AOS: вместо массива структур `{phase, freq, amp}` — структура массивов `{phases[], freqs[], amps[]}`. Это позволяет векторизацию across voices.

#### 2.2.3 Пороговые эвристики для триггера JIT

Баланс между временем компиляции и выполнения критичен. **Адаптивная стратегия:**

- Начальная фаза: консервативные пороги, быстрая обратная связь
- Стабилизация: повышение порогов для частых паттернов
- Деградация: понижение качества (без JIT) при перегрузке CPU

Профилирование в реальном времени отслеживает: время выполнения каждого блока, частоту кэш-промахов, использование SIMD. При превышении бюджета — **graceful degradation**: упрощение алгоритмов, понижение разрешения, пропуск некритичных эффектов.

### 2.3 Интеграция с низкоуровневым аудио

#### 2.3.1 Буферизация и обработка по блокам

Аудиодрайверы работают с блоками фиксированного размера. **Компромисс latency vs. эффективность:**

| Размер буфера | Задержка (48 кГц) | Применение |
|-------------|------------------|-----------|
| 16 сэмплов | 0.33 мс | Профессиональный мониторинг, инструменты |
| 64 сэмпла | 1.33 мс | Интерактивная работа, live coding |
| 256 сэмплов | 5.33 мс | Микширование, пост-обработка |
| 1024 сэмпла | 21.3 мс | Фоновое воспроизведение, экономия энергии |

Linen использует **двойную/тройную буферизацию**: пока один буфер воспроизводится, другой заполняется VM, третий готовится в фоне. Для записи — аналогично.

Внутри блока обработка **векторизована**; между блоками — синхронизация состояния (фазы осцилляторов, содержимое линий задержки).

#### 2.3.2 Интерфейс с аудиодрайверами (JACK, ALSA, CoreAudio)

Абстракция `AudioBackend` скрывает платформенные детали. Реализации:

| Бэкенд | Платформа | Особенности |
|--------|----------|------------|
| **JACK** | Linux, macOS | Низкая latency, межприложенная маршрутизация, предпочтительный |
| **ALSA** | Linux | Нативный, fallback когда JACK недоступен |
| **CoreAudio** | macOS | Нативный, интеграция с системным микшером |
| **WASAPI (Exclusive)** | Windows | Низкая latency, обход системного микшера |
| **PortAudio** | Кроссплатформенный | Fallback, разработка и тестирование |

JACK особенно подходит для Linen: его модель портов и соединений естественно отображается на графовую структуру языка. Порты Linen-клиента становятся JACK-портами, соединения управляются внешне.

#### 2.3.3 Гарантии низкой задержки

Жёсткие real-time гарантии невозможны в общем случае, но минимизация рисков:

| Мера | Реализация |
|-----|-----------|
| Приоритет потока | `SCHED_FIFO`, real-time priority 80+ |
| Блокировка памяти | `mlockall()`, предотвращение выгрузки страниц |
| Изоляция CPU | `isolcpus`, dedicated cores для аудио |
| Избегание системных вызовов | Предвыделение всех ресурсов, lock-free структуры |
| Мониторинг | Встроенный профилировщик, статистика dead-line'ов |

**Измерение и диагностика**: отслеживание min/max/среднего/99-го перцентиля времени выполнения колбэка. Предупреждение при превышении 50% бюджета, критическая ошибка при пропуске dead-line.

## 3. Стандартная Библиотека

### 3.1 Базовые операции

#### 3.1.1 Арифметика сигналов

Поточечные операции над сигналами — фундамент микширования и модуляции. Все операции подняты из скаляров через аппликативный функтор.

| Операция | Тип | Описание |
|---------|-----|---------|
| `(+)` | `Num a ⇒ Signal a → Signal a → Signal a` | Микширование (с возможным клиппингом) |
| `(*)` | `Num a ⇒ Signal a → Signal a → Signal a` | Амплитудная модуляция, кольцевая модуляция |
| `(/)` | `Fractional a ⇒ Signal a → Signal a → Signal a` | Нормализация, делительная модуляция |
| `abs` | `Num a ⇒ Signal a → Signal a` | Выпрямление, детектор огибающей |
| `signum` | `Num a ⇒ Signal a → Signal a` | Квантование по уровню, hard clipping |
| `clip` | `Ord a ⇒ a → a → Signal a → Signal a` | Ограничение диапазона (soft/hard limiter) |

Нелинейные операции для искажения: `tanh` (soft clipping), `distort` (waveshaping с произвольной кривой), `fold` (foldback distortion).

#### 3.1.2 Фильтры: ФНЧ, ФВЧ, полосовые

**Биквадратный фильтр (biquad)** — универсальная структура IIR-фильтра второго порядка. Передаточная функция:

$$H(z) = \frac{b_0 + b_1 z^{-1} + b_2 z^{-2}}{1 + a_1 z^{-1} + a_2 z^{-2}}$$

Коэффициенты вычисляются из параметров для различных типов:

| Тип | Параметры | Применение |
|-----|----------|-----------|
| Lowpass | Частота среза, Q | Удаление высоких частот, тёплые звуки |
| Highpass | Частота среза, Q | Удаление низких частот, ясность |
| Bandpass | Центр, Q | Выделение полосы, формантный фильтр |
| Notch | Центр, Q | Удаление гудения, резонансов |
| Peak | Центр, Q, gain | Параметрический эквалайзер |
| Lowshelf/Highshelf | Частота, gain, S | Полочные фильтры, тональная коррекция |

Реализация использует **прямую форму II** с двумя элементами состояния. Для высоких Q и низких частот — **state-variable filter** или **ladder filter** с лучшей численной устойчивостью.

Каскадирование биквадратов создаёт фильтры высоких порядков (Butterworth, Chebyshev, Elliptic) с крутым спадом. Функции `butterworth`, `chebyshev1/2`, `elliptic` генерируют каскады заданного порядка и характеристик.

#### 3.1.3 Модуляция: AM, FM, кольцевая

| Тип | Формула | Спектральный эффект |
|-----|--------|---------------------|
| **AM** | `(1 + m·modulator) · carrier` | Несущая + две боковые полосы |
| **RM (кольцевая)** | `modulator · carrier` | Только боковые полосы, без несущей |
| **FM** | `sin(2π · ∫(f_c + Δf·modulator)dt)` | Бесконечное число боковых полос, богатый спектр |
| **PM** | `sin(2π·f_c·t + β·modulator)` | Эквивалентно FM при малых индексах, проще в реализации |

FM-синтез в стиле Yamaha DX7: 6 операторов, 32 алгоритма соединения, обратная связь. Тип `DX7Algorithm = Graph Operator Feedback`, где `Operator = { ratio, level, envelope, velocitySensitivity }`.

### 3.2 Ввод-Вывод

#### 3.2.1 Чтение и запись WAV

Формат WAV (RIFF/WAVE) — несжатый PCM, приоритет для начальной реализации. Поддерживаемые конфигурации:

| Параметр | Значения |
|---------|---------|
| Каналы | 1 (mono), 2 (stereo), многоканальный (WAVEFORMATEXTENSIBLE) |
| Разрядность | 16-bit int, 24-bit int, 32-bit float, 64-bit float |
| Частота дискретизации | 44100, 48000, 88200, 96000, 192000 Гц |

API:
```haskell
readWav      ∷ FilePath → Audio (AudioBuffer Float)
writeWav     ∷ FilePath → AudioBuffer Float → Audio ()
readWavStream ∷ FilePath → Audio (Stream (AudioBuffer Float))  -- потоковое
```

Расширение для больших файлов: **RF64** (EBU Tech 3306), позволяющий файлы > 4 ГБ.

#### 3.2.2 Импорт и экспорт MIDI

MIDI — дискретный протокол, естественно отображающийся на события Linen. Поддержка:

| Сообщение | Отображение в Linen |
|----------|---------------------|
| Note On/Off | `Event (Note, Velocity, Channel)` |
| Control Change | `Event (CC, Value, Channel)` → `Behavior Float` через `midiCC` |
| Program Change | `Event (Program, Channel)` |
| Pitch Bend | `Event Int14` → интерполяция в `Behavior Float` |
| Clock/Song Position | Синхронизация транспорта |

Функции:
```haskell
midiIn       ∷ Audio (Event MidiMessage)
midiOut      ∷ Event MidiMessage → Audio ()
midiNote     ∷ Channel → Event (Note, Velocity)  -- фильтр по каналу
midiCC       ∷ Channel → CC → Behavior Float     -- непрерывный контроллер
syncToMidiClock ∷ Event MidiMessage → Behavior BPM  -- извлечение темпа
```

Экспорт MIDI-файлов (SMF) для офлайн-рендеринга паттернов.

#### 3.2.3 Реальное время: захват и воспроизведение

Интеграция с аудиоинтерфейсом через абстракцию `AudioDevice`:

```haskell
data AudioDevice = AudioDevice {
  sampleRate    ∷ Rate,
  bufferSize    ∷ Int,
  numInputs     ∷ Int,
  numOutputs    ∷ Int,
  processCallback ∷ AudioBuffer Float → IO (AudioBuffer Float)
}

openDevice   ∷ DeviceConfig → Audio AudioDevice
closeDevice  ∷ AudioDevice → Audio ()
startStream  ∷ AudioDevice → IO ()
stopStream   ∷ AudioDevice → IO ()
```

**Low-latency monitoring**: прямой путь вход→выход с минимальной обработкой для мониторинга исполнителя.

### 3.3 Утилиты времени

#### 3.3.1 Синхронизация по BPM

Транспорт — единственный источник мутабельного состояния. Операции:

| Функция | Тип | Описание |
|--------|-----|---------|
| `setBPM` | `BPM → Audio ()` | Установка темпа |
| `tapTempo` | `Event () → Behavior BPM` | Определение темпа по нажатиям |
| `rampBPM` | `BPM → Time → Audio ()` | Плавное изменение темпа |
| `syncToTransport` | `Behavior a → Behavior (Synced a)` | Привязка к музыкальной сетке |

#### 3.3.2 Полиметрия и сложные ритмы

Поддержка **нескольких одновременных временных шкал**:

```haskell
polyrhythm ∷ [(BPM, TimeSignature, Signal a)] → Signal [a]
euclidean  ∷ Int → Int → Pattern Bool  -- алгоритм Бьорклунда
```

Полиритмика: 3 против 2, 4 против 3 и т.д. Полиметрия: различные размеры такта с общей долей.


## 4. Экосистема Библиотек

### 4.1 Музыкальные абстракции

#### 4.1.1 Ноты, гаммы, аккорды

**Типы для музыкальной теории:**

| Тип | Определение | Операции |
|-----|------------|---------|
| `Pitch` | `PitchClass × Octave` | Транспозиция, интервалы |
| `PitchClass` | `C | Cis | D | Dis | ...` | Повышение, понижение |
| `Interval` | `Quality × Number` | Сложение, инверсия |
| `Scale` | `[Interval]` | Построение из тоники, степени |
| `Chord` | `[Interval]` | Инверсии, расширения, замены |

Гаммы: мажор, натуральный/гармонический/мелодический минор, пентатоники, блюз, модусы, экзотические (восточные, цыганские).

Аккорды: трезвучия, септаккорды, расширенные (9, 11, 13), альтерированные.

#### 4.1.2 Инструменты и патчи

**Патч** — параметризованная конфигурация синтезатора:

```haskell
data Patch = Patch {
  name        ∷ String,
  parameters  ∷ Map String (Behavior Float),
  voice       ∷ VoiceConfig → Signal Float,
  polyphony   ∷ Int
}

type VoiceConfig = { pitch ∷ Pitch, velocity ∷ Float, gate ∷ Event () }
```

Полифония через **voice allocation**: динамическое распределение голосов, stealing при перегрузке, ротация для равномерного износа.

#### 4.1.3 Синтезаторы: субтрактивный, FM, гранулярный

| Тип | Принцип | Ключевые параметры |
|-----|--------|-------------------|
| **Субтрактивный** | Богатый источник + фильтрация | Осциллятор, cutoff, resonance, envelope |
| **FM** | Модуляция частоты, сложные спектры | Алгоритм, ratios, indices, envelopes |
| **Гранулярный** | Микро-фрагменты сэмпла с перекрытием | Grain size, density, position, pitch |
| **Аддитивный** | Сумма синусоид | Частоты, амплитуды, фазы частичных |
| **Физическое моделирование** | Модель акустической системы | Параметры материала, возбуждение |

### 4.2 Эффекты и обработка

#### 4.2.1 Пространственные эффекты: реверберация, дилэй

| Эффект | Реализация | Параметры |
|--------|-----------|----------|
| **Stereo delay** | Кольцевой буфер с отводами | Time L/R, feedback, mix, ping-pong |
| **Multitap delay** | Несколько отводов с разными задержками | Taps[], levels, panning |
| **Convolution reverb** | FFT-свёртка с импульсной характеристикой | IR file, predelay, wet/dry |
| **Algorithmic reverb** | Сеть аллпасов и комб-фильтров | Room size, decay, damping, diffusion |
| **Shimmer** | Pitch-shifted feedback в ревербе | Shift amount, feedback, blend |

#### 4.2.2 Динамическая обработка: компрессия, лимитер

| Эффект | Принцип | Ключевые параметры |
|--------|--------|-------------------|
| **Compressor** | Уменьшение gain при превышении threshold | Threshold, ratio, attack, release, knee |
| **Limiter** | Жёсткое ограничение пиков | Ceiling, lookahead, release |
| **Expander/Gate** | Уменьшение gain ниже threshold | Threshold, ratio, range, hold |
| **De-esser** | Компрессия только в полосе частот | Frequency, Q, threshold |
| **Multiband** | Независимая обработка полос | Crossovers, per-band parameters |

Sidechain: внешний ключ для ducking, pumping эффектов.

#### 4.2.3 Дисторшн и сатурация

| Тип | Характеристика | Применение |
|-----|---------------|-----------|
| **Soft clip (tanh)** | Плавное ограничение | Тёплое насыщение, ламповый звук |
| **Hard clip** | Резкое ограничение | Агрессивный distortion |
| **Waveshaping** | Произвольная функция | Специфические тембры |
| **Bit crusher** | Квантование амплитуды и времени | Lo-fi, ретро-эффекты |
| **Sample rate reducer** | Понижение частоты дискретизации | Алиасинг как эффект |

## 5. Интеграция с DAW

### 5.1 Архитектура плагинов

#### 5.1.1 Linen как встроенный язык скриптов

Linen встраивается в DAW как **скриптовый движок** для генерации и обработки аудио. Режимы интеграции:

| Режим | Описание | Применение |
|------|-----------|-----------|
| **Track effect** | Обработка аудио трека | Пользовательские фильтры, эффекты |
| **Instrument** | Генерация MIDI→аудио | Синтезаторы, сэмплеры |
| **MIDI effect** | Преобразование MIDI | Арпеджиаторы, генераторы |
| **Clip/Region generator** | Создание контента | Алгоритмические партии |

API взаимодействия с хостом: параметры автоматизации, tempo, time signature, transport state, sidechain inputs.

#### 5.1.2 Живое кодирование (live coding) в DAW

Интеграция REPL с DAW: редактирование кода с немедленным применением. Ключевые особенности:

- **Разделение экрана**: редактор Linen + стандартный интерфейс DAW
- **Визуальная обратная связь**: осциллограф, спектроанализатор, сигнальный граф
- **Запись сессий**: сохранение истории редактирования как нелинейная композиция

#### 5.1.3 Горячая перезагрузка без потери состояния

Механизм IFRP применяется к DAW-контексту: изменение кода инструмента сохраняет:
- Фазы всех осцилляторов
- Содержимое линий задержки и ревербераторов
- Позиции огибающих и секвенсоров
- Состояние фильтров и модуляторов

### 5.2 Пользовательский интерфейс

#### 5.2.1 Визуализация сигнальных графов

Автоматическая генерация графа из кода: узлы — примитивы и пользовательские функции, рёбра — потоки данных. Интерактивность: раскрытие/свёртывание, редактирование параметров, точки останова для отладки.

#### 5.2.2 Редакторы патчей и модулярные среды

Визуальный редактор как альтернатива коду: drag-and-drop примитивов, соединение проводами, редактирование параметров. Биекция с кодом: любой патч можно экспортировать в Linen и обратно.

#### 5.2.3 Интеграция параметров DAW с переменными Linen

Автоматизация: параметры DAW отображаются на `Behavior Float` в Linen. Обратное направление: экспорт `Behavior Float` как автоматизируемый параметр. Синхронизация через pull (запрос значения) или push (callback при изменении).

## 6. Документация для Разработки

### 6.1 Спецификация языка

#### 6.1.1 Грамматика и синтаксис

Linen использует синтаксис, вдохновлённый Haskell и OCaml:

```
program     ::= module* declaration*

declaration ::= type | let | letrec | extern

type        ::= "type" con "=" typeExpr
typeExpr    ::= typeAtom ("->" typeExpr)?
typeAtom    ::= con | var | "(" typeExpr ")"

let         ::= "let" var param* ":" type "=" expr
letrec      ::= "let" "rec" (var param* ":" type "=" expr)+

expr        ::= atom | app | infix | lambda | if | match
atom        ::= var | con | num | string | "(" expr ")"
app         ::= expr atom+
infix       ::= expr op expr
lambda      ::= "\" param+ "->" expr
if          ::= "if" expr "then" expr "else" expr
match       ::= "match" expr "{" (pattern "->" expr)* "}"

pattern     ::= var | con pattern* | "_" | literal
```

Ключевые слова: `let`, `let rec`, `type`, `match`, `if`, `then`, `else`, `in`, `forall`, `behavior`, `event`, `signal`.

#### 6.1.2 Семантика типов и вывода типов

Алгоритм W для HM с расширениями. Правила для FRP-примитивов:

```
Γ ⊢ e : Behavior τ
─────────────────────
Γ ⊢ sample e : Event τ' → Event τ

Γ ⊢ e₁ : Event (τ → τ')    Γ ⊢ e₂ : τ
──────────────────────────────────────
Γ ⊢ accum e₁ e₂ : Behavior τ'
```

#### 6.1.3 Операционная семантика FRP-примитивов

Малый шаг для сигнальных функций:

```
⟨sf, s⟩ ──a──→ ⟨b, s'⟩
────────────────────────
⟨arr f, s⟩ ──a──→ ⟨f a, s⟩

⟨sf₁, s₁⟩ ──a──→ ⟨b, s₁'⟩    ⟨sf₂, s₂⟩ ──b──→ ⟨c, s₂'⟩
────────────────────────────────────────────────────────
⟨sf₁ >>> sf₂, (s₁, s₂)⟩ ──a──→ ⟨c, (s₁', s₂')⟩
```

### 6.2 Архитектурные документы

#### 6.2.1 Описание VM и байткода

Формат байткода, layout объектов в памяти, calling conventions, исключения и обработка ошибок.

#### 6.2.2 Протокол JIT-интерфейса

API для подключения бэкендов компиляции: `JITBackend { compile :: Bytecode -> MachineCode, optimize :: IR -> IR }`.

#### 6.2.3 Модель памяти и сборка мусора в реальном времени

Формальная модель: регионы, линейные типы, взаимодействие с аудиопотоком.

### 6.3 Руководства по расширению

#### 6.3.1 Создание библиотек на Linen

Структура пакета: `linen.toml` (манифест), `src/`, `tests/`, `examples/`. Публикация в центральный репозиторий.

#### 6.3.2 FFI для интеграции внешних DSP-библиотек

Биндинги к C/C++: `extern "C" { ... }`, marshalling данных, управление временем жизни.

#### 6.3.3 Оптимизация критичных участков

Профилирование, идентификация горячих путей, ручная векторизация, использование `@jit` аннотаций.

### 7.3 Рабочий процесс

#### 7.3.1 Сборка интерпретатора и VM

```bash
# Клонирование
git clone https://github.com/linen-lang/linen.git
cd linen

# Установка зависимостей
opam install . --deps-only

# Сборка
dune build

```

#### 7.3.2 Запуск тестовых аудиопрограмм

```bash
# Интерактивный REPL
linen repl

# Выполнение файла
linen run examples/sine_wave.ln

# С компиляцией в байткод
linen compile -o output.bc examples/sine_wave.ln

# С JIT-компиляцией
linen run --jit examples/complex_patch.ln
```

Пример тестовой программы:

```linen
-- sine_wave.ln
let sampleRate = 48000.0
let freq = 440.0

let sineWave : Behavior Float =
  let phase = integral (freq / sampleRate) in
  sin (2.0 * pi * phase)

let main : Audio () =
  audioOut (sineWave * 0.5)
```
